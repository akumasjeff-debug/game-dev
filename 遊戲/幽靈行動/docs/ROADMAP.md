# 幽靈行動 — Demo 發布路線圖

版本：1.0 | 制定日：2026-06-21 | 制定者：策略顧問

---

## 一、目前已有什麼（可以玩的）

P0 + P1 + P2 骨架完成，玩家目前能體驗到以下完整流程：

**戰場層面（P0 + P1）**
- 小隊 5 人自動沿 waypoint 路徑推進
- 遭遇決策點時畫面暫停，彈出選項面板（直衝/靜悄/炸彈，補給箱三選一，岔路左右）
- 選擇後繼續推進，岔路即時切換路徑
- 5 張職業卡大招 HUD，點擊即時施放（盾兵護盾、突擊手攻擊 buff、狙擊手秒殺標記、醫療兵回血、爆破手 AOE）
- 盾兵 Lv.3 在房間決策自動加入「舉盾突入」選項（條件解鎖已驗證）
- 勝利/失敗結算畫面 + 重試按鈕

**Meta 層面（P2 骨架）**
- 基地場景框架：任務板 UI + 4 格角色陣容選擇
- 離線金幣計算系統（計算邏輯已實作）
- JSON 資料驅動（角色設定、決策點設定已讀取）
- 本機存檔（ConfigFile）

**已通過的技術驗證：** headless 測試 exit 0，無 ERROR，CanvasLayer UI 修正完畢。

**目前缺的（玩起來有明顯破洞）：** 6 選 4 陣容選擇尚未與戰場連通、角色 JSON 只有 5 職業（偵察手缺）、P2 基地到戰場的完整流程串接不完整、所有畫面都是佔位圖形（無精靈圖、無 isometric 場景）。

## 二、Demo 最小可行版本定義

### 「完全玩不起來」的缺失（必須有）

以下任何一個缺失，玩家就無法完整走完一局：

| 缺失項目 | 說明 |
|---------|------|
| 基地 → 陣容選 → 任務 → 結算 → 回基地 完整閉環 | 目前各段是孤立場景，沒有連通 |
| 6 選 4 陣容選擇與戰場角色同步 | 玩家選的陣容必須影響戰場 HUD 和大招 |
| 偵察手加入 characters.json | 6 職業系統才完整，否則「6 選 4」無從選起 |
| 至少 1 張可完整跑完的關卡（含 Boss 房） | 目前地圖只有 waypoint 佔位，需要真正的任務終點與 Boss 事件 |
| 數值平衡緊急修正（3 個嚴重問題） | 依 BALANCE_REVIEW.md，狙擊手 CD 50 秒 / 偵察手被動失效 / 爆破手+狙擊手無限秒殺，會在前 10 分鐘就讓玩家得出「這遊戲不好玩」的結論 |

### 「體驗不完整但不卡死」的缺失（Demo 可以沒有）

| 缺失項目 | 說明 |
|---------|------|
| Lv.6 / Lv.9 高階決策選項 | Demo 可以只做到 Lv.1–3 |
| 抽卡系統 | 用固定預設陣容代替 |
| 主線故事敘事 / 任務文字背景 | 可以只有標題 |
| Meshy.ai 3D 模型 | 彩色方塊或 2D 精靈可以代替 |
| 音效完整 | 佔位符 WAV（現有合成音效）夠用 |
| 升等系統完整介面 | Demo 可以只給預設 Lv.3 角色，跳過升等流程 |
| 多關卡 / 任務板多任務 | 1 個任務夠了 |

### Demo 最小標準（定義）

> 玩家進入遊戲 → 看到基地畫面 → 選擇 4 名隊員 → 進入任務 → 小隊自動推進 → 遭遇 3–4 個決策點並做選擇 → 抵達 Boss 房 → 觸發 Boss 事件決策 → 成功/失敗結算 → 回到基地。完整一局時間約 5 分鐘，可重複遊玩 3 局以上不感到重複。

## 三、發布 Demo 需要的剩餘工作

### 技術面（最多工作量在這裡）

| 任務 | 優先度 | 估時 |
|------|--------|------|
| 打通基地 → 任務選擇 → 戰場 → 結算 → 回基地 完整 SceneManager 流程 | 必須 | 2–3 天 |
| 6 選 4 陣容選擇 UI（基地頁面完整化），選擇結果傳入戰場初始化 | 必須 | 1–2 天 |
| 偵察手加入 characters.json，P2 格式對接完成 | 必須 | 0.5 天 |
| 設計並實作 1 張完整關卡（含 waypoint、3–4 房間事件、岔路、Boss 決策點） | 必須 | 2–3 天 |
| Boss 事件邏輯（進入 Boss 房前的決策：直衝 / 迂迴 / 引蛇出洞，以及 Boss 觸發勝利） | 必須 | 1 天 |
| 套用 BALANCE_REVIEW 的 3 個嚴重數值修正 | 必須 | 0.5 天 |
| 任務獎勵系統（成功給金幣/票券，基地離線金幣完整閉環） | 必須 | 1 天 |
| HUD 修正：從 5 人固定改為 4 人動態（與 GDD v2.1「6 選 4」一致） | 必須 | 0.5 天 |
| HTML5 匯出 + Playwright 本地截圖驗證 | 必須 | 0.5 天 |

**技術面合計估時：9–12 天（含測試）**

### 內容面

Demo 最小需要：
- **1 個任務**：1 條路線，5–6 個決策點（3 個房間 + 1 個岔路 + 1 個補給箱 + 1 個 Boss 房前事件）
- **6 個職業**：Lv.1–3 可玩（偵察手需補完）
- **2–3 個特殊事件文字**（只用目前已有的「房間/補給箱/岔路」即可）

Demo 不需要：
- 主線故事敘事、隱藏事件觸發（那是升等體驗，Demo 不做到 Lv.9）
- 多任務類型（支線）
- 計時炸彈、人質等高複雜特殊事件

### 美術面

**可以用方塊代替發布嗎？可以，但有一個條件。**

- 角色：彩色方塊（每個職業用不同顏色區分）可以接受。玩家需要「一眼知道誰是誰」——如果顏色夠鮮明，方塊完全可以發 Demo。
- 場景：灰色平面 + 房間邊框輪廓（2D 俯視或簡易 Isometric 格線）可以接受。
- UI：基地任務板、陣容選擇、HUD 需要可讀性（目前 CanvasLayer 結構已對），只要文字清楚就夠。
- **不可以缺少的：** 每個職業的顏色標記必須一致（HUD 卡片顏色 = 戰場方塊顏色），否則玩家無法建立認知。

**結論：方塊美術完全可以發 Demo，但需要明確的職業顏色規範。**

建議職業顏色：
- 盾兵：藍色
- 醫療兵：白/綠
- 突擊手：橙色
- 狙擊手：黃色
- 爆破手：紅色
- 偵察手：紫色

### 音效面

**佔位符 WAV 夠用嗎？夠，不是理想但夠。**

GDD 定義的音效方向（槍聲、決策暫停提示音、大招施放回饋）對 Demo 體驗有直接影響，但正弦波合成音已能夠提供：
- 決策點觸發的「停止感」：一個清晰的中頻提示音足夠
- 大招施放回饋：不同頻率可區分職業

**Demo 必須有的 WAV 優先順序：**
1. 決策點觸發音（最重要，玩家需要聽到）
2. 大招施放音（各職業共用一個版本即可）
3. 勝利/失敗音效

BGM 可以暫缺，或用一首循環的低頻環境音佔位。

## 四、路線圖（時間估算）

### 里程碑 1：流程閉環（約 1.5 週）
**目標：從基地到結算的完整一局能跑完，不依賴手動切場景。**

- 實作 SceneManager / 場景切換系統
- 基地任務板：點擊「開始任務」進入 6 選 4 陣容選擇，確認後啟動戰場
- 戰場初始化從選擇的陣容讀取（4 名職業，HUD 對應顯示）
- 戰場結算後點擊「回基地」返回基地，金幣/票券入帳
- 偵察手補入 characters.json，P1/P2 格式整合完畢
- HUD 從 5 人固定改為 4 人動態

**驗收標準：** 能連續跑 3 局，每次選不同陣容，結算金幣正確入帳。

---

### 里程碑 2：關卡完整 + 數值修正（約 1 週）
**目標：有一個設計完整、數值感好的關卡可以玩。**

- 設計並實作完整 Demo 關卡（參考 GDD 十四章 MVP 定義）：
  - 5–6 個決策點，含岔路 + 補給箱 + Boss 房前事件
  - 終點觸發任務成功
- 套用 BALANCE_REVIEW 三個嚴重修正：
  - 狙擊手 Lv.1 大招 CD 50 秒 → 35 秒
  - 狙擊手大招條件改為「目標 HP < 25%」才能瞬殺
  - 偵察手 Lv.1 被動改為「未知房間自動揭露敵人數量」
- 明確限定「每職業只能帶一名」規則
- 關卡測試：玩 10 局確認可玩性（不是技術測試，是「好不好玩」測試）

**驗收標準：** 5 分鐘內完成一局，勝率約 50–60%（Lv.1 陣容），連玩 3 局不覺得無聊。

---

### 里程碑 3：Demo 打磨 + 發布準備（約 0.5 週）
**目標：可以讓外部玩家試玩而不尷尬的品質。**

- 職業顏色規範落地（方塊 + HUD 卡片顏色一致）
- 決策點觸發音 + 大招施放音 WAV 到位
- 新手第一局的引導（最少：第一個決策點旁邊加一句說明文字，告訴玩家「點選下方選項做選擇」）
- HTML5 匯出 + Playwright 截圖驗證通過
- itch.io 頁面建立：標題、1 張截圖、100 字描述

**驗收標準：** 匯出檔在瀏覽器開啟，截圖有畫面，一局流程走完無 crash。

---

### 預估發布時程

| 里程碑 | 目標完成 | 累計時間 |
|--------|---------|---------|
| 里程碑 1：流程閉環 | 約 10 天後 | 第 10 天 |
| 里程碑 2：關卡完整 + 數值修正 | 再 7 天 | 第 17 天 |
| 里程碑 3：Demo 打磨 + 發布 | 再 3–4 天 | 第 20–21 天 |

**Demo 最快發布時間：從今天起算約 3 週（2026-07-12 前後）。**

---

### 發布平台建議

**首選：itch.io HTML5**

理由：
1. 零安裝門檻，玩家直接在瀏覽器遊玩，是 Demo 驗證最快的方式
2. itch.io 有足夠的獨立遊戲玩家受眾，可以收集初步回饋
3. Godot 4 HTML5 匯出流程已有前例（台灣熱炒王有驗證），技術風險可控
4. 發布成本零，不需要 App Store 審核等待

**不建議直接上 Android/iOS（Demo 階段）：**
- App Store / Google Play 審核時間 1–2 週，拖延回饋速度
- Demo 品質的方塊美術不適合在商店留下第一印象
- 等正式版本才上架

## 五、最大風險

### 風險 1：場景切換 + 資料傳遞的整合複雜度（高風險）

**描述：** 目前基地和戰場是分開的孤立場景。要把「6 選 4 陣容選擇結果」正確傳入戰場初始化（決定 HUD 顯示哪 4 個職業、每個職業的等級和大招 CD）需要一個穩健的資料傳遞架構。Godot 4 的 SceneManager + Autoload 整合如果架構不對，P1/P2 已有的邏輯可能需要大量重構。

**規避方式：**
- 優先設計 GameManager 的 session 資料結構（selected_party: Array[職業]), 讓基地和戰場都能讀這個全域狀態
- 里程碑 1 的前 2 天先用最簡單的 print() 驗證資料流通，確認架構可行再做 UI

---

### 風險 2：偵察手被動重設計的複雜度（中風險）

**描述：** BALANCE_REVIEW 建議把偵察手 Lv.1 被動改為「進入未知房間自動揭露敵人數量」。這個設計需要：(1) 決策面板能顯示額外資訊行、(2) 房間決策點資料結構要支援「敵人數量」欄位、(3) 偵察手的 Lv.1 狀態檢查邏輯。是一個跨系統的改動。

**規避方式：**
- 如果整合複雜度高，Demo 版允許妥協：偵察手 Lv.1 被動改為「決策面板上所有選項各自附加一行簡短情報（格式：難 / 中 / 易）」，這樣只改 DecisionPanel 的顯示邏輯，不動其他系統。先讓偵察手「看起來有在做事」，正式版再做完整揭露。

---

### 風險 3：Isometric 場景的等待期（低風險，但時間陷阱）

**描述：** GDD 定義的 Isometric 斜角俯視視覺是賣點，但目前所有原型都是 2D 俯視佔位。如果在 Demo 前想做一版正式 Isometric 場景（Camera3D 斜角固定 + 低多邊形建築），Meshy.ai 生成 + 匯入 + 調整可能花費 2–5 天，且 Godot 4 的 3D isometric 有 lighting/shadow/layering 的陷阱問題。

**規避方式：**
- Demo 版明確放棄 Isometric 3D，採用 2D 俯視或 2.5D 預渲染貼圖（靜態背景）。Isometric 視覺是正式版目標，不是 Demo 目標。這個決策可以省下 3–5 天，換來關卡內容和數值的更多時間。
